# 汉化实现方案

本文是汉化"怎么做"的**结论**（去掉探索过程；踩坑历程见 [`HISTORY.md`](HISTORY.md)）。
三大挑战：① CHR 字库一时刻只能寻址 256 tile，中文 1472 唯一字远超；② 8px 字号清晰度；
③ 演出（折行/分页/名字/立绘）由结构块驱动，回写须忠实保留。

构建入口：[`../src/build_ch1_mb.py`](../src/build_ch1_mb.py)（第一章多 bank 定版）。

## 一、字模：8×8 原版机制 + 像素字体

- 引擎沿用**原版 8×8**：上格保持 `0xFE` 空白、下格填中文字模，**每字仅 1 tile**
  （密度 ~238 字/4KB-bank）。字竖直贴下排，与原版假名位置一致。
- 字体用 [Fusion Pixel Font](https://github.com/TakWolf/fusion-pixel-font)（手工 hint 像素字体，
  OFL 授权，随仓库于 [`../fonts/`](../fonts/)）。矢量字硬缩到 8px 会糊成一团；fusion-8px 原生点阵
  8 宽也清晰。字模生成见 `reinsert_full.glyph8x8`。
- 8×16 / 真 16×16 更清晰但需改渲染，权衡见 [`ENGINE.md`](ENGINE.md#两套-emit)，列为后续 polish。

## 二、字库 bank：每场景独立 + 单点 asm 钩子

全部汇编改动收敛到**极少数单点等长 patch**，其余全是数据回写。

1. **扩 CHR / PRG → 1MB**：给中文字库和 block 流腾出全新空间（`rom[4]=64; rom[5]=128`）。
   ★关键坑：扩 PRG 后上电 `$5117=$FF`→末 bank 映射 `$E000`，新区是空的→复位向量读 0 崩，
   故把原 PRG 末 8KB（向量+固定代码）拷到新 PRG 末 8KB。
2. **B 命门（对话框字库 bank）**：NMI `$EB7F`（file `0x7EB8F`）进入时寄存器 A 已 = `$0450`
   （背景 CG bank），把 `LDA $0451; STA $045F`（`AD5104 8D5F04`）改成
   **`ORA #$80; STA $045F; NOP`**（`0980 8D5F04 EA`）→ 对话框字库 bank = **`$0450 | 0x80`**。
3. **每场景独立字库**：于是每个 CG 场景（`$0450`）自动映射到扩 CHR 的 `(cg|0x80)` bank，
   **白嫖游戏现成的场景切换，无需查表/新代码/bank63 空间**。装箱粒度 = 场景（同场景句必须同 bank；
   实测每场景 < 213 字）。每 bank = bank0 拷贝（保边框/空白/图标）+ 该场景字模。

## 三、数据回写

- **装箱架构**（`reinsert_full.py` / `build_ch1_mb.py`）：唯一字形 → 共享 block（全篇复用），
  按场景 first-fit 装箱。码池从 `0x2A` 起（保留 `0x10-0x29` 图标/方向键/♥♫★/数字菜单序号），
  预留 `0xD0-0xDF` 边框、`0xFE` 空白。
- **结构化管线**（`structio.py`）：折行 `/`、设置块 `{sHH}`、空隙块 `{gHH}`、控制码 `~XXXX~`
  全部保留为 token，翻译在其上进行（折行按中文短语边界人工放置），回写忠实重建。演出
  （换行/分页/颜色/打字机速度/"等待翻页▽"）全对。全 2781 句结构 token 与日文 0 处不一致。
- **空间布局**（避撞存活数据）：block 的 1B text 串 → 真空闲 `TEXT_FREE=(0x73604, 0x74000)`；
  句子 block 流 → 新增扩 PRG `BLK_FREE=(0x80010, 0x100010)`。
  ★`0x74000-0x75B00` 是 bank58 立绘/头像指针表、`0x76000+` 是 bank59 启动代码，**绝不能覆盖**
  （build 有断言 `rom[0x74000:0x75B00]==原版` 守护）。

## 四、密语 / CONTINUE 存档系统

存档密语 = 一句日语短句（第一章示例 "えりな もう おきなよ"，句 154 在剧情里口述教给玩家）。
CONTINUE 屏用**全假名选择格**输入，故**密语必须保原版假名，绝不能翻译**（否则玩家学到中文却要
按假名输入 → 存档功能废）。修复（`build_ch1_mb.py`）：

- **bank0 保持纯净**（253 码非空的原版满字库）：密语/存档屏（场景 00）用它渲染假名。
- 绿字改用独立 `GREEN_BANK=0xE0`（不再灌进 bank0）。
- 场景 00 的句（`gui_scene==0`）不改写；系统句（22/15/17/19/20…）引用的 block 全部 RESERVED，
  不被当空闲挪用。
- **KANA2CODE**：译文里保留的原版假名走原始 tile 码（码从码池保留）→ 句 154 密语显示为假名，
  玩家照假名形输入即可。
- 实测：CONTINUE 屏假名格 + 独白与原版逐字节一致；密语显示假名；绿字中文；场景顶部 CG 完好。

## 五、当前状态与已知限制

**第一章可玩版**：118 句主线 + 41 回访；立绘/启动/向量/头像脸/名字左端全正常；密语存档闭环。

| 限制 | 说明 | 根治方向 |
|---|---|---|
| 句 187（梓聊父亲）| 情感戏、24 字溢出单场景 bank | 多 bank 第二维度，或留日文 |
| 命令名跨场景乱码 | 查看/交谈等字仅在恰好出现的场景入库 | 命令名当固定字，每 bank 必注入 |
| 深话题 / 回访乱码 | 句在非其字库场景显示 | 多 bank 或按显示场景重复打包 |
| 区框装饰显中文 | 说话人小头像边框走对话区 bank | GUI+debugger 看框实际 tile/bank，或 asm 分屏精细化 |
| 真 16×16 | 8px 是硬物理限制 | 重写 bank62 打字机子系统（有风险） |

（`$80+场景`）/ B bank（`$A0+场景`）间选，让热点场景（海边、查米近景 >190 字）容纳全量翻译。
